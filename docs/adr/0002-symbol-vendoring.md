# ADR 0002 â€” Vendoring KiCad symbols for catalog parts without stock library entries

- **Status**: Accepted
- **Date**: 2026-05-21
- **Builds on**: [ADR 0001 â€” Radix UI Primitives + existing tokens](0001-ui-primitives.md), PR #38 which added the `kicadLibId` field and stock-library symbol placement.

## Context

PR #38 wired the bundle exporter to place stock KiCad symbols for the four catalog parts that exist in KiCad 7+ stock libraries (STM32F405RG, STM32H743ZI, RP2040, W25Q128JV). The four remaining catalog parts have **no stock KiCad symbol** and are currently skipped at export time:

| Part | Role | Stock KiCad symbol? |
|---|---|---|
| ICM-42688-P (TDK IMU) | Sensor | No |
| BMP388 (Bosch barometer) | Sensor | No |
| CH340K (USB-UART bridge) | Interface | No |
| Tag-Connect 6-pin debug header | Connector | Maybe (`Connector.kicad_sym`) |

A user exporting a project that includes any of these gets a valid-but-incomplete schematic. To close the loop we need a vendoring strategy.

## Decision

Vendor the missing symbols in this repo under `catalog/symbols/`, reference them from catalog JSON via a new optional `vendoredSymbolPath` field, and have the bundle exporter inline the symbol body into the schematic's `(lib_symbols ...)` block at export time. KiCad opens the schematic with no external library lookup needed.

When and only when a vendored source is not viable for a specific part:

- Synthesize a minimum-viable symbol (pins + properties only, no graphics) from the datasheet pinout. KiCad opens it as a bounding-box placeholder; users can refine in the Symbol Editor.

The runtime exporter only **reads** vendored files; it never **writes** to `catalog/symbols/`. Adding a symbol is a code change.

## License + attribution

KiCad's stock library license + SnapEDA's standard license both carry a **design exception** that waives attribution within the generated design (the `.kicad_sch` itself). Attribution is only required for **redistribution of the symbol files**.

Required:

- `catalog/symbols/LICENSE.md` documents each source + license + a verbatim copy of the design exception text.
- Per-file header comment inside each vendored `.kicad_sym` retains the original source line if present.

Not required (but ship anyway for hygiene):

- An optional `;; Generated by KiForge â€” embeds symbols from KiCad official libs (CC-BY-SA 4.0) and SnapEDA (CC-BY-SA 4.0 + DE 1.0). See â€¦/LICENSE.md.` comment header in every exported `.kicad_sch`.

## Alternatives considered

| Approach | Why not |
|---|---|
| Runtime download from SnapEDA / Ultra Librarian | Adds network dependency + auth + per-call rate limit. Tauri capabilities would need `http:` plugin. Defeats offline-first. |
| Synthesize symbols from datasheet pinouts hand-coded in TS | Maintenance burden â€” every part requires a TS module mirroring the symbol body. JSON + raw `.kicad_sym` files are simpler and the format is already battle-tested by KiCad. |
| Defer to "user provides their own symbol library" | Beta blocker. The smoke flow expects "open the bundle â†’ see the parts". A blank or placeholder symbol is worse UX than a slightly-imperfect SnapEDA one. |
| Ultra Librarian as a source | License terms for redistribution in a Tauri app are not clearly documented. SnapEDA's design exception is explicit. |
| Embed symbol body inline in catalog JSON (`vendoredSymbolBody: "(symbol ..."`) | Bigger JSON files, lost diff readability vs a real `.kicad_sym`, no way for a contributor to open + edit the symbol in KiCad's Symbol Editor. |
| Inline-embed only properties + pins (skip vendored files entirely, synthesize per part) | Acceptable fallback (KiCad shows a grey rectangle placeholder), but ships ugly output. Use only when no licensed source exists. |

## What changes in the codebase

- `catalog/symbols/` (new directory)
  - `LICENSE.md` documenting both source licenses + design exceptions verbatim.
  - `kicad-official/TagConnect.kicad_sym`
  - `snapeda/BMP388.kicad_sym`
  - `snapeda/ICM-42688-P.kicad_sym`
  - `snapeda/CH340K.kicad_sym`
- `src/types/domain.ts`: `ComponentCatalogEntry` gains optional `vendoredSymbolPath?: string` (path relative to repo root, e.g. `catalog/symbols/snapeda/BMP388.kicad_sym`).
- `src/features/catalog/catalog.ts`: Zod string field, no regex (it's a file path; just non-empty).
- `catalog/components/{bmp388,icm-42688-p,ch340k,tag-connect}.json`: each gets `vendoredSymbolPath` plus a `kicadLibId` matching the symbol name inside the vendored file (e.g. `BMP388:BMP388`). The library prefix is arbitrary â€” what matters is that the symbol name in the vendored `.kicad_sym` matches the symbol part of the `lib_id` string.
- `src/features/catalog/load-vendored-symbol.ts` (new): pure function that reads a vendored `.kicad_sym` file at module evaluation time via Vite's `import.meta.glob({ as: "raw", eager: true })` so the symbol bodies bundle into the frontend asset. Returns a `Map<libId, sExprString>`.
- `src/features/export/build-kicad-bundle.ts`: accepts the symbol-body map via a new `vendoredSymbols?: Map<string, SList>` option. When set, the exporter parses each body into an `SList`, dedupes by symbol name, and pushes them into the schematic's `(lib_symbols ...)` block before the placed-symbol nodes.
- `src/features/export/build-kicad-bundle.test.ts`: covers (a) a placed symbol whose `libId` matches a vendored entry surfaces the body in `lib_symbols`, (b) duplicate placed symbols only land once, (c) absent `vendoredSymbols` map = today's behavior (empty `lib_symbols`).
- `docs/beta-checklist.md`: smoke flow step 12 updated â€” "open in KiCad, expect placed controller + flash + BMP388 + IMU + CH340K + Tag-Connect symbols all render with real shapes (not bounding-box placeholders)."

Not part of this ADR (separate slices, each its own PR):

- The vendored-symbol read path itself (`load-vendored-symbol.ts` + bundle wire) â€” **Phase 1**.
- Sourcing + committing each vendored `.kicad_sym` â€” **Phase 2**, one PR per part so each can be reviewed against its source URL.

## What does NOT change

- `kicadLibId` semantics from PR #38: a part with **only** a `kicadLibId` (no `vendoredSymbolPath`) keeps relying on the user's installed KiCad library. The four stock-library parts stay as-is.
- The runtime never downloads symbols. Network is not added to Tauri capabilities.
- Rust side. Symbol vendoring is frontend-only â€” the Tauri write command keeps writing whatever bytes the frontend sends; it has no notion of `lib_symbols`.

## Adoption plan

**Phase 1 â€” exporter wire (1 PR, mine or engineer)**

- Add `vendoredSymbolPath` to schema + Zod.
- Add `load-vendored-symbol.ts` with `import.meta.glob` on `catalog/symbols/**/*.kicad_sym`.
- Wire `vendoredSymbols` option into `build-kicad-bundle.ts`; parse + dedupe + inject into `(lib_symbols ...)`.
- Tests as above.
- Ships with **zero vendored files** so behavior on main stays identical. Path is now ready to receive files.

**Phase 2 â€” vendor each part (4 PRs, one per part, parallelizable to engineers)**

Per part: commit the vendored `.kicad_sym` under the right subdir, update `catalog/symbols/LICENSE.md`, set `vendoredSymbolPath` + `kicadLibId` in the catalog JSON, smoke that an export including the part renders the symbol in KiCad 8. One self-contained PR each. Order doesn't matter; safest to start with Tag-Connect (KiCad stock source, smallest license footprint) to prove the pipeline.

**Phase 3 â€” beta-readiness sweep (1 PR)**

- Update `docs/beta-checklist.md` smoke step 12.
- Add the optional `;; Generated by KiForge â€” embeds symbols from â€¦` header to every exported `.kicad_sch`.

## Consequences

**Positive**

- Beta schematics open in KiCad with every catalog part rendered, no missing-symbol pop-ups.
- Each vendored source is auditable: `.kicad_sym` is plain s-expr, diffable, openable in KiCad's Symbol Editor.
- No new runtime dependency, no network access, no Tauri capability change.

**Negative**

- Repo grows by ~50-200 KB per `.kicad_sym` file. Acceptable.
- SnapEDA's 10-file ceiling is now part of our contract. Tracking: with 3 SnapEDA files in Phase 2, we have 7 left for catalog growth before the next ADR needs to revisit (e.g. switch to Ultra Librarian or hand-synthesis).
- A vendored symbol can fall out of sync with the part's silicon (silkscreen rename, pin shuffle on an errata revision). Mitigation: PR template question â€” "did you verify the vendored symbol against the latest datasheet?".
